1. Introduction

1.1 Purpose

This Software Requirements Specification (SRS) defines the complete functional and non-functional requirements for the Sales Order Management System (SOMS) developed by Atreya SoftTech Pvt. Ltd. The document serves as the authoritative reference for design, development, testing, and project acceptance.

SOMS is a mobile-first web application that digitises and streamlines the field-sales order capture and production scheduling workflow. It replaces manual entry processes and provides real-time visibility from order creation through to delivery completion.

1.2 Scope

The system covers the following functional domains:

  • Secure authentication via mobile number and 6-digit PIN
  • Role-based access control — two roles: Admin and Sales User
  • Sales order entry, submission, and personal order tracking by Sales Users
  • Centralised order management dashboard for Admins (customer-wise and item-wise views)
  • Production queue management with batch-level report generation
  • Order completion workflow with vehicle and notes capture
  • Report generation, download, and order archival by Admin

Out of scope for v1.0: Inventory management, payment processing, ERP integration, third-party logistics connectivity.

2. Overall Description

2.1 Product Perspective

SOMS is a standalone, cloud-hosted web application accessible from any modern mobile browser without requiring a native app installation. It communicates with a RESTful API backend and a relational database, with extension points for future ERP and CRM integration.

2.2 Product Functions — Summary

#FunctionActorDescription

F-02Order EntrySales UserCreate and submit sales orders with customer and item details
F-03My Orders ViewSales UserView personal order list; update completed orders with delivery info
F-04Admin DashboardAdminView all orders in customer-wise and item-wise tabs
F-05Production SchedulingAdminMark item-wise order groups as 'In Production'; generate batch reports
F-06Order CompletionSales UserMark own order complete; add vehicle details and notes
F-07Completed Order MgmtAdminView completed orders; download reports; delete orders

2.3 User Classes

2.3.1 Sales User

Field sales representatives who interact with customers and record sales transactions. They are expected to be comfortable with smartphones. The UI prioritises simplicity and speed. Sales Users can only see and manage their own orders.

2.3.2 Administrator

Administrators manage the overall order lifecycle, coordinate production, and maintain data hygiene. They have full visibility across all Sales Users and all orders. All Admin accounts share identical privileges in v1.0.

2.4 Operating Environment

  • Primary: Mobile browser (Chrome/Safari on iOS 16+, Android 10+)
  • Secondary: Desktop browsers (Chrome 110+, Firefox 110+, Edge 110+)
  • Network: 4G/LTE or Wi-Fi; graceful degradation on 3G
  • Hosting: Cloud-hosted, containerised deployment (AWS/Azure)

3. Specific Functional Requirements

3.1 Authentication — FR-AUTH

Req IDRequirementPriority
FR-AUTH-01The system shall present a login screen as the default landing page for all unauthenticated sessions.High
FR-AUTH-02The login screen shall contain a 10-digit mobile number input field and a 6-digit numeric PIN input field.High
FR-AUTH-03The PIN field shall mask entered digits, displaying them as dots or asterisks.High
FR-AUTH-04On successful authentication, the system shall issue a secure JWT session token with a configurable expiry (default 8 hours).High
FR-AUTH-05On authentication failure, the system shall display "Invalid mobile number or PIN" without indicating which field is incorrect (prevents enumeration attacks).High
FR-AUTH-06After 5 consecutive failed login attempts, the account shall be locked for 15 minutes.High
FR-AUTH-07After successful login, Sales Users are redirected to Order Entry; Admins are redirected to the Admin Dashboard.High
FR-AUTH-08A 'Logout' option shall be accessible from all authenticated screens.Medium
FR-AUTH-09Sessions shall be invalidated server-side on logout; the client token must be discarded.High
FR-AUTH-10The mobile number field shall accept only 10 numeric digits (no country code prefix).Medium

3.2 Order Entry — FR-ORDER

Req IDRequirementPriority
FR-ORDER-01Authenticated Sales Users shall be able to access an Order Entry form from their home screen.High
FR-ORDER-02The Order Entry form shall contain the following mandatory fields: Customer Name (text), Customer Mobile Number (10-digit), Item Code (searchable), Quantity (positive integer).High
FR-ORDER-03All four fields shall be validated client-side and server-side before submission.High
FR-ORDER-04The system shall automatically record the submitting Sales User's ID and the UTC timestamp — these fields are not editable.High
FR-ORDER-05On successful submission, the system shall show a confirmation message and clear the form for the next entry.Medium
FR-ORDER-06A Sales User shall be able to submit multiple orders in the same session without logging out.High
FR-ORDER-07Quantity must be a positive integer greater than zero. Zero or negative values shall be rejected with an inline validation message.High
FR-ORDER-08Customer Mobile Number shall be validated as exactly 10 numeric digits.Medium
FR-ORDER-09The Item Code field shall be a searchable dropdown of pre-loaded valid item codes (not free text).Medium

3.3 My Orders View — FR-MYORDERS

Req IDRequirementPriority
FR-MYORDERS-01The system shall display a 'My Orders' screen showing only orders submitted by the currently logged-in Sales User.High
FR-MYORDERS-02Orders shall be displayed in ascending chronological order (oldest first — first billed, first displayed) by default.Medium
FR-MYORDERS-03Each order row shall display: Customer Name, Item Code, Quantity, Submission Date/Time, and current Order Status.High
FR-MYORDERS-04The view shall support filtering by Order Status (All / Pending / In Production / Completed).Medium
FR-MYORDERS-05Completed orders shall be visually distinguished (e.g., greyed out or with a distinct badge).Medium
FR-MYORDERS-06A Sales User shall not be able to edit or delete a submitted order. Only the completion workflow can modify it.High

3.4 Admin Dashboard — FR-ADMIN

3.4.1 Customer-Wise Tab

Req IDRequirementPriority
FR-ADMIN-01The Customer-Wise tab shall group and display all active orders grouped by Customer Name.High
FR-ADMIN-02Each customer group shall be collapsible/expandable to show or hide its individual orders.Medium
FR-ADMIN-03Each order entry shall display: Order ID, Customer Mobile, Item Code, Quantity, Sales User Name, Submission Date, and Order Status.High
FR-ADMIN-04The tab shall support searching by Customer Name or Customer Mobile Number.Medium
FR-ADMIN-05Completed orders shall be included in the view with a 'Completed' status badge and filterable by status.Medium

3.4.2 Item-Wise Tab

Req IDRequirementPriority
FR-ADMIN-06The Item-Wise tab shall group and display all active orders grouped by Item Code.High
FR-ADMIN-07Each item group shall display the total aggregated quantity for that Item Code across all included orders.High
FR-ADMIN-08Each item group shall list individual orders with Customer Name, Quantity, Sales User, and Submission Date.High
FR-ADMIN-09Each item group in 'Pending' state shall display a 'Set to Production' button.High
FR-ADMIN-10⚠️ When a new order arrives for an Item Code that already has an active production batch, the new order shall appear as a separate pending group and must NOT be merged into the existing in-production batch.High
FR-ADMIN-11The Item-Wise tab shall support filtering by Order Status (All / Pending / In Production / Completed).Medium

3.5 Production Scheduling — FR-PROD

Req IDRequirementPriority
FR-PROD-01An Admin shall be able to click 'Set to Production' on an item group to transition all included pending orders to 'In Production' status as a single batch.High
FR-PROD-02The system shall require a confirmation prompt before transitioning orders to 'In Production' status.Medium
FR-PROD-03On confirming 'Set to Production', the system shall immediately generate a Production Batch Report.High
FR-PROD-04The Production Batch Report shall be downloadable as a PDF from within the application.High
FR-PROD-05The Production Batch Report shall contain: Report Date/Time, Batch ID, Item Code, Total Quantity, and a table of all orders in the batch (Order ID, Customer Name, Quantity, Sales User, Submission Date).High
FR-PROD-06⚠️ Once an item group is moved to 'In Production', new orders for the same Item Code shall form a new separate pending group — they shall not be added to the existing production batch.High
FR-PROD-07The status change to 'In Production' shall be logged with the Admin User ID and the timestamp of the action.High
FR-PROD-08The 'Set to Production' action shall be irreversible; there is no 'Revert to Pending' function in v1.0.High

3.6 Order Completion — FR-COMPLETE

Req IDRequirementPriority
FR-COMPLETE-01A Sales User shall be able to mark one of their own orders as 'Completed' from the My Orders view.High
FR-COMPLETE-02The completion action shall open a form requiring: Vehicle Number (mandatory) and Notes (optional free-text).High
FR-COMPLETE-03Vehicle Number shall be validated as non-empty before the completion form can be submitted.High
FR-COMPLETE-04On submitting the completion form, the order status shall change to 'Completed' and the completion timestamp shall be recorded.High
FR-COMPLETE-05A Sales User may only complete orders belonging to themselves; they cannot complete another Sales User's orders.High
FR-COMPLETE-06An order that has not been moved to 'In Production' can still be completed (e.g., immediate small orders).Medium
FR-COMPLETE-07Once marked 'Completed', no further edits are possible by the Sales User.High

3.7 Completed Order Management — FR-MGMT

Req IDRequirementPriority
FR-MGMT-01The Admin Dashboard shall include a 'Completed Orders' section to view all orders with status 'Completed'.High
FR-MGMT-02Each completed order record shall display: Order ID, Customer Name, Customer Mobile, Item Code, Quantity, Sales User, Submission Date, Completion Date, Vehicle Number, and Notes.High
FR-MGMT-03An Admin shall be able to download a Completed Order Report for selected orders in PDF format.High
FR-MGMT-04The Completed Order Report shall include all fields from FR-MGMT-02 plus the Admin who downloaded the report and the download timestamp.Medium
FR-MGMT-05An Admin shall be able to permanently delete a completed order record after downloading the report.High
FR-MGMT-06The system shall display a confirmation dialog before any delete action, clearly stating it is permanent and irreversible.High
FR-MGMT-07Deleted orders shall be soft-deleted in the database (flagged, not physically removed) to preserve an audit trail. They shall not appear in any UI view after deletion.High
FR-MGMT-08An Admin shall be able to select multiple completed orders and delete them in bulk.Medium

4. Android Application

4.1 Platform Overview

The Sales Order Management System (SOMS) is developed as a native Android application. It is designed to be installed and run on Android smartphones and tablets, providing an optimised mobile experience for field sales representatives and administrators.

4.2 Target Platform & Compatibility

AttributeDetails
PlatformAndroid (Native)
Minimum SDKAndroid 10 (API Level 29)
Target SDKAndroid 14 (API Level 34)
Supported DevicesSmartphones and tablets with screen sizes from 5" and above
DistributionDirect APK distribution
Architecture SupportARM64-v8a, armeabi-v7a

4.3 Android-Specific Features

  • The application shall be built as a native Android app, ensuring full access to device capabilities and a native look-and-feel consistent with Android design guidelines (Material Design).
  • The app shall support offline data caching, allowing Sales Users to view previously loaded orders when network connectivity is temporarily unavailable.
  • Push notifications shall be supported to alert Sales Users and Admins of order status changes.
  • The app shall handle Android lifecycle events (screen rotation, backgrounding, process death) gracefully, preserving form data and user state.
  • The app shall request only the minimum necessary Android permissions required for its functionality (e.g., INTERNET, ACCESS_NETWORK_STATE).

4.4 Installation & Deployment

The SOMS Android application shall be distributed as a direct APK provided to users for installation. Updates shall be managed through the distribution of updated APK files. The minimum storage space required for installation shall not exceed 50 MB.

5. User Interface Requirements

5.1 Login Screen

  • Fields: Mobile Number (numeric input) + PIN (masked numeric)
  • Primary action: full-width 'Login' button
  • Error state: inline error below the form on failure
  • Lockout state: disabled form with countdown timer showing minutes remaining
  • Branding: Atreya SoftTech logo and app name at top

5.2 Sales User — Order Entry

  • Bottom tab bar with 'New Order' (active) and 'My Orders'
  • Form field order: Customer Name → Customer Mobile → Item Code (searchable) → Quantity
  • Real-time inline validation on blur for each field
  • Submit button disabled until all fields are valid
  • Success: toast/snackbar with order reference ID; form resets automatically

5.3 Sales User — My Orders

  • Scrollable list of order cards, oldest first (first billed, first displayed)
  • Card content: Customer Name, Item Code, Quantity, Date, Status badge
  • Status badges: Pending (yellow) · In Production (blue) · Completed (green)
  • 'Mark Complete' button visible only on Pending or In Production orders
  • Completion modal: Vehicle Number (required) + Notes (optional) + Confirm button
  • Filter chips: All / Pending / In Production / Completed

5.4 Admin Dashboard

  • Top tab bar: Customer Wise | Item Wise
  • Customer Wise: Accordion groups by customer name; order count badge; expanded state shows order table
  • Item Wise: Accordion groups by item code; total quantity badge; 'Set to Production' button on pending groups
  • 'Set to Production' flow: Confirmation modal → success toast with batch ID → report download prompt
  • Search bar on both tabs

5.5 Admin — Completed Orders

  • Full-width table with all completed order fields
  • Row-level checkbox selection for bulk operations
  • Toolbar: 'Download Report' and 'Delete' buttons activate when 1+ rows selected
  • Delete confirmation modal: lists selected order IDs, warns action is permanent
  • Report download: triggers PDF generation and browser file download
